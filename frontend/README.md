# Frontend

Lighthouse UI — the single-page inspection surface for the Chorus Phase 1 vertical slice.

## What is here

A standalone React + Vite + TypeScript app. Stack:

- React 19 + Vite 8 + TypeScript 5.8
- TanStack Router (file-based) and TanStack Query
- Tailwind v4 via `@tailwindcss/vite`, configured CSS-first through tokens under `src/design-system/tokens/`
- Radix primitives, Lucide icons, Sonner toasts
- Vitest + Testing Library, Playwright for E2E

The Dense design family, tokens, and contracts are vendored in-tree under `src/design-system/`. There is no workspace linkage to any sibling project.

## Run it

```bash
cd frontend
npm install
npm run dev
```

The dev server defaults to `http://localhost:5173`. `/api` is proxied to the BFF (`http://localhost:8000` by default; override with `VITE_API_PROXY_TARGET`).

## Scripts

| Script | Purpose |
|---|---|
| `npm run dev` | Vite dev server with HMR. |
| `npm run build` | Type-check (`tsc -b`) and produce a production bundle in `dist/`. |
| `npm run preview` | Serve the production bundle locally. |
| `npm run typecheck` | `tsc --noEmit`. |
| `npm run lint` | Same as `typecheck` for now (no ESLint config). |
| `npm run fmt` / `npm run fmt:check` | Prettier write / verify. |
| `npm test` | Vitest in run mode. |
| `npm run test:watch` | Vitest in watch mode. |
| `npm run test:e2e` | Playwright tests against a started dev server. |

## Routes

File-based, under `src/routes/`:

| Path | Purpose |
|---|---|
| `/` | Workflow run list. |
| `/workflows/$workflowId` | Workflow detail — timeline, decision trail, tool verdicts. |
| `/decision-trail` | Filterable agent decision table across all runs. |
| `/tool-verdicts` | Filterable Tool Gateway verdict table. |
| `/registry` | Read-only agent registry. |
| `/grants` | Read-only tool grants. |
| `/routing` | Read-only model routing rules. |
| `/eval` | Eval harness fixture results. |

The route tree (`src/routeTree.gen.ts`) is regenerated by the TanStack Router Vite plugin on `npm run dev` or `npm run build`. A committed seed file is provided so a fresh clone type-checks before the dev server has run.

## Design constraints

The Lighthouse UI is a dense, data-first inspection surface for an architecture review, not a marketing page.

- No card layouts for primary content.
- No decorative hero sections.
- Tables, timelines, plain text, filters, read-only detail views.
- UI state must survive refresh and reconnect by reading projections from the BFF; SSE is a progress hint, not the source of truth.
- All mutating action authority sits with the Tool Gateway and Temporal workflows. The UI never originates writes.

`src/design-system/` includes `Card` and `Hero` from the source family for completeness, but the Phase 1 routes deliberately do not use them.

## Design system

`src/design-system/` is a wholesale, in-tree copy of the Dense family plus tokens and contracts from a sibling design-system project. See `src/design-system/README.md` for layout. The barrel at `src/design-system/index.ts` re-exports the dense components and contract types.

## Backend integration

Stub fixtures live in `src/api/fixtures.ts` and feed every route through TanStack Query. Replace `queryFn`s with calls into `src/api/client.ts` (`apiGet`) once the matching BFF endpoints land. SSE progress events flow through `src/api/sse.ts` — treat them as cache invalidation hints, not as state.
